# Data Dictionary — bluestock_mf.db

This documents the final Day 2 deliverable: a SQLite star-schema database
built from 10 real source CSVs covering Indian mutual fund NAV, investor
transactions, performance, AUM, portfolio holdings, and benchmark data.

Every number on this page was generated by actually running
`data_cleaning.py` then `load_to_sqlite.py` against the 10 real CSVs and
querying the resulting `bluestock_mf.db` — nothing here is estimated.

## 2. Cleaning rules applied (`data_cleaning.py`)

| File | Rules |
|---|---|
| `nav_history` | Parse dates; coerce NAV to numeric; non-positive NAV treated as missing; drop duplicate (amfi_code, date) rows; **reindex each fund onto a continuous daily calendar and forward-fill** so every weekend/holiday gets an explicit, ffilled row (this is why row count grows from 46,000 to 64,320); drop any row still unfillable. |
| `investor_transactions` | Standardize `transaction_type` to SIP / Lumpsum / Redemption via a case-insensitive lookup (flags, doesn't silently drop, anything unrecognized); coerce `amount` to numeric, drop rows ≤ 0 or null; parse dates; flag (don't drop) any `kyc_status` outside {Verified, Pending, Rejected}; drop exact duplicate rows. |
| `scheme_performance` | Coerce every return-style column (`return_1yr_pct`, `return_3yr_pct`, `return_5yr_pct`, `benchmark_3yr_pct`) to numeric; flag (don't drop) `expense_ratio` outside the realistic 0.05%–2.5% TER band; flag any return with magnitude > 100% as suspicious; drop exact duplicates. |
| `portfolio_holdings` | Parse `portfolio_date`; drop rows with non-positive `weight_pct`; **sanity-check that each fund's holdings sum to ~100%** and warns if any fund is outside 95–105%; drop duplicates. |
| `benchmark_indices` | Parse dates; coerce `close_value` to numeric; drop non-positive values; drop duplicate (date, index_name) rows. |
| All other files (3, 4, 5, 6) | Generic clean only: drop exact duplicate rows, parse any column with "date" in its name. No domain-specific validation since they're outside the star schema's current scope (see §6). |

On this real dataset, every file came back **completely clean** except
`nav_history` (which gained rows from the weekend/holiday forward-fill, by
design) — zero invalid amounts, zero bad KYC values, zero duplicate rows
anywhere.

## 3. Star schema overview

Two dimensions (`dim_fund`, `dim_date`), six facts
(`fact_nav`, `fact_transactions`, `fact_performance`, `fact_aum`,
`fact_portfolio_holdings`, `fact_benchmark_prices`), all linked through
`amfi_code` and/or `date_id`. Full DDL in `schema.sql`.

### dim_fund (40 rows)
One row per AMFI scheme code.

| Column | Type | Business definition |
|---|---|---|
| amfi_code | INTEGER (PK) | Unique AMFI scheme code. Codes can be reassigned if a scheme closes/merges — don't trust the code alone, cross-check `scheme_name`. |
| fund_house | TEXT | AMC managing the fund, e.g. "SBI Mutual Fund". |
| scheme_name | TEXT | Full scheme/plan name, e.g. "SBI Bluechip Fund - Regular Plan - Growth". |
| category | TEXT | High-level category, e.g. "Equity". |
| sub_category | TEXT | SEBI sub-category, e.g. "Large Cap". |
| risk_grade | TEXT | Riskometer label, e.g. "Moderate". (Sourced from `risk_category` in the raw file, renamed to match this schema.) |

**Note:** `01_fund_master.csv` actually has 15 columns — `plan`,
`launch_date`, `benchmark`, `expense_ratio_pct`, `exit_load_pct`,
`min_sip_amount`, `min_lumpsum_amount`, `fund_manager`, and
`sebi_category_code` are cleaned and sitting in
`data/processed/01_fund_master.csv`, but **not currently loaded** into
`dim_fund` — there's no column for them yet. Add them to `schema.sql`'s
`CREATE TABLE dim_fund` and to the `keep` list in `load_to_sqlite.py` if
you want them in the database.

### dim_date (1,609 rows)
One row per calendar date appearing in any fact table — built
automatically from the union of every date across all loaded sources,
not from a source file.

| Column | Type | Business definition |
|---|---|---|
| date_id | INTEGER (PK) | Surrogate key, `YYYYMMDD` as an integer (e.g. 20240115). |
| full_date | TEXT | ISO date string 'YYYY-MM-DD'. |
| year / quarter / month / day | INTEGER | Calendar parts. |
| month_name / day_of_week | TEXT | e.g. "January" / "Monday". |
| is_weekend | INTEGER | 1 if Saturday/Sunday, else 0. |

### fact_nav (64,320 rows)
Daily NAV per fund, gap-filled (see §2).

| Column | Type | Business definition |
|---|---|---|
| nav_id | INTEGER (PK) | Surrogate row key. |
| amfi_code | INTEGER (FK -> dim_fund) | Fund this NAV belongs to. |
| date_id | INTEGER (FK -> dim_date) | Date of this NAV. |
| nav | REAL | Net Asset Value, ₹ per unit. Always > 0. |

### fact_transactions (32,778 rows)
One row per investor transaction — **all 13 original columns preserved**,
including investor demographics.

| Column | Type | Business definition |
|---|---|---|
| transaction_id | INTEGER (PK) | Surrogate row key. |
| amfi_code | INTEGER (FK) | Fund invested in/redeemed from. |
| date_id | INTEGER (FK) | Transaction date. |
| investor_id | TEXT | Investor identifier. |
| transaction_type | TEXT | Standardized to SIP / Lumpsum / Redemption. |
| amount | REAL | Transaction amount, ₹. Always > 0. |
| kyc_status | TEXT | Verified / Pending / Rejected (30,146 / 2,632 / 0 in this dataset). |
| state, city, city_tier | TEXT | Investor location; city_tier is T30/B30 (Top-30 vs Beyond-30 cities, the standard AMFI classification). |
| age_group, gender | TEXT | Investor demographics. |
| annual_income_lakh | REAL | Self-reported annual income, ₹ lakh. |
| payment_mode | TEXT | UPI / NetBanking / Cheque / Mandate / NEFT etc. |

### fact_performance (40 rows)
**Single point-in-time snapshot**, not a time series — `07_scheme_performance.csv`
has no date column, so every row is tagged with the date the pipeline was
run (`load_to_sqlite.py` prints this date each run). If you re-run the
pipeline on a different day, this snapshot date will change.

| Column | Type | Business definition |
|---|---|---|
| performance_id | INTEGER (PK) | Surrogate row key. |
| amfi_code | INTEGER (FK) | Fund this snapshot belongs to. |
| date_id | INTEGER (FK) | The pipeline run date (snapshot proxy, not a real "as of" date from the source). |
| return_1y / return_3y / return_5y | REAL | Trailing annualized return, %. |
| expense_ratio | REAL | Total Expense Ratio, %. |
| alpha, beta | REAL | Standard risk-adjusted performance measures vs benchmark. |
| sharpe_ratio, sortino_ratio | REAL | Risk-adjusted return ratios. |
| std_dev_ann_pct | REAL | Annualized standard deviation of returns, %. |
| max_drawdown_pct | REAL | Largest peak-to-trough decline, %. |
| benchmark_3yr_pct | REAL | The fund's benchmark index's own 3-year return, % — for comparison against `return_3y`. |
| morningstar_rating | INTEGER | Star rating, 1–5. |

### fact_aum (40 rows)
AUM per fund, same snapshot-date caveat as `fact_performance` — this is
sourced from the `aum_crore` column inside `07_scheme_performance.csv`,
**not** from `03_aum_by_fund_house.csv` (that file is a different grain —
by fund house, not by individual scheme — see §6).

| Column | Type | Business definition |
|---|---|---|
| aum_id | INTEGER (PK) | Surrogate row key. |
| amfi_code | INTEGER (FK) | Fund this AUM belongs to. |
| date_id | INTEGER (FK) | Snapshot date (see fact_performance note). |
| aum_cr | REAL | Assets Under Management, ₹ crore. |

### fact_portfolio_holdings (322 rows)
Stocks held by each fund, as of `portfolio_date`.

| Column | Type | Business definition |
|---|---|---|
| holding_id | INTEGER (PK) | Surrogate row key. |
| amfi_code | INTEGER (FK) | Fund holding this stock. |
| date_id | INTEGER (FK) | Portfolio snapshot date. |
| stock_symbol | TEXT | NSE ticker, e.g. "HDFCBANK". |
| stock_name | TEXT | Full company name. |
| sector | TEXT | Industry sector, e.g. "Banking". |
| weight_pct | REAL | % of fund's portfolio in this stock. Per-fund weights are validated to sum to ~100% during cleaning. |
| market_value_cr | REAL | Holding's market value, ₹ crore. |
| current_price_inr | REAL | Stock's price at snapshot time, ₹. |

### fact_benchmark_prices (8,050 rows)
Daily closing values for benchmark indices (NIFTY50, NIFTY100, etc.),
used to compare a fund's NAV growth against its benchmark (see query 12).

| Column | Type | Business definition |
|---|---|---|
| benchmark_id | INTEGER (PK) | Surrogate row key. |
| date_id | INTEGER (FK) | Date of this closing value. |
| index_name | TEXT | e.g. "NIFTY50". |
| close_value | REAL | Index closing value that day. |

## 4. Indexes

`(amfi_code, date_id)` composite indexes exist on every fact table that
has both columns, plus a `(date_id, index_name)` index on
`fact_benchmark_prices` — these back the fund+date join pattern almost
every query in `queries.sql` uses.

## 5. The 13 analytical queries (`queries.sql`)

| # | Query | Verified result on real data |
|---|---|---|
| 1 | Top 5 funds by AUM (latest snapshot) | Mirae Asset Emerging Bluechip leads at ₹49,046 cr |
| 2 | Average NAV per month per fund | 2,120 fund-month rows |
| 3 | SIP year-over-year growth | 2024 → 2025 SIP inflow fell ~58% (see caveat below) |
| 4 | Transactions by state | 12 states, Punjab highest by total amount |
| 5 | Funds with expense_ratio < 1% | 14 funds qualify |
| 6 | Total AUM by fund house | Nippon India MF leads |
| 7 | Top 5 funds by 1-year return | ABSL Small Cap tops at 24.93% |
| 8 | KYC status breakdown | 30,146 Verified, 2,632 Pending, 0 Rejected |
| 9 | Monthly transaction volume/amount trend | 17 month-rows |
| 10 | Funds with zero transactions | 0 — every fund in the master has at least one transaction |
| 11 | Top sectors by portfolio weight | Consumer Goods, Diversified lead |
| 12 | Fund NAV vs benchmark over time | 1,150 daily comparison rows for SBI Bluechip vs NIFTY100 |
| 13 | Most widely-held stocks across funds | Bharti Airtel held by 15 funds |

**Caveat on query 3 (SIP YoY growth):** this 32,778-row transaction file
only spans Jan 2024–Dec 2025 (2 calendar years), so the year-over-year
comparison is a single 2024-vs-2025 data point, not a multi-year trend.
That's a property of the source data's date range, not a query bug.

## 6. Known scope gaps — not yet in the database

Four real source files are cleaned (sitting in `data/processed/`) but
**not loaded into `bluestock_mf.db`**, because they're industry-level
aggregates at a different grain than the per-fund star schema:

- `03_aum_by_fund_house.csv` — AUM by fund house *and quarter* (not by individual scheme)
- `04_monthly_sip_inflows.csv` — industry-wide monthly SIP totals
- `05_category_inflows.csv` — monthly net inflows by fund category
- `06_industry_folio_count.csv` — industry-wide monthly folio counts

These would need either a separate `fact_industry_monthly` table (no
`amfi_code`, just `dim_date` + category/fund_house dimensions) or a
decision to leave them as supplementary CSVs for direct pandas analysis
outside the database. Flagging this explicitly rather than silently
leaving them out undocumented.

## 7. Reproducing this database

```cmd
python data_cleaning.py
python load_to_sqlite.py
```
Both scripts print row-count verification as they run — compare that
output against the tables in §1 and §3 above to confirm your run matches.
